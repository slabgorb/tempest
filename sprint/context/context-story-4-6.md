## Technical Approach

This story wires the persistence seam (4-4: `shell/storage.ts`) to the in-memory high-score table (4-3: `core/state.ts` and `core/highscore.ts`) at two points in the shell bootstrap and loop:

1. **At boot:** Call `loadHighScores()` in `src/main.ts` after `initialState()` is created, then mutate the initial state's `highScoreTable` field before passing it to `createLoop()`. This seeds persisted scores into the in-memory table, so they appear on the attract screen immediately on page load.

2. **On commit:** When a high-score entry is committed (player enters initials), the state machine (4-3) inserts the new entry into the table and transitions `'highscore' → 'attract'`. Wire the `onModeChange` hook (4-5: already added to `createLoop`) to detect this transition: when `oldMode === 'highscore'` and the transition happens, call `saveHighScores(loop.getState().highScoreTable)` to persist the updated table.

**Confirmation:** The high-score table is only mutated by the state machine during the 'highscore' mode, and the only mode transition out of 'highscore' is to 'attract' (after the entry is committed). So a single `onModeChange` trigger at `oldMode === 'highscore'` is sufficient.

**Scope (shell only):**
- Modify `src/main.ts` (bootstrap + loop wiring) — this is the only file that needs changes.
- Do NOT modify `src/core/state.ts`, `src/core/highscore.ts`, or `src/shell/storage.ts` (those are done in 4-3, 4-4).
- Pure core remains pure (no persistence imports, no `window`/`localStorage`).

**Optional / out of scope:**
- Stamping a `date` on saved entries is optional (core never inserts one). Keeping in-memory and persisted tables identical (no date) is simplest.

**Testing:** No unit tests for shell bootstrap in this project. Verify via `npx tsc --noEmit` (typecheck) + `npm run build` (production build), and manually: load game → make a high score → enter initials → reload page → score persists on attract/gameover screen.

## Scope
- In scope: Boot-time seed of persisted high scores; on-commit save via `onModeChange` hook.
- Out of scope: Changes to `core/` or `shell/storage.ts`; unit tests for bootstrap.

## Acceptance Criteria

1. **AC1 — Boot loads persisted table:** On page load, `loadHighScores()` is called and the initial `GameState.highScoreTable` is seeded with persisted entries before the loop starts. Scores appear on the attract screen immediately.

2. **AC2 — Commit persists table:** When mode transitions from `'highscore'` to `'attract'` (entry committed), `saveHighScores(loop.getState().highScoreTable)` is called to persist the updated table.

3. **AC3 — Persistence survives page reload:** After entering a high-score entry and reloading the page, the new entry appears in the high-score table on the attract or gameover screen.

4. **AC4 — No core changes:** `src/core/state.ts`, `src/core/highscore.ts`, and `src/core/sim.ts` remain unchanged; core stays pure.

5. **AC5 — No storage-seam changes:** `src/shell/storage.ts` (loadHighScores, saveHighScores) remains unchanged; the seam is complete.

6. **AC6 — Typecheck and build clean:** `npx tsc --noEmit` and `npm run build` both exit 0 with no errors or warnings.
